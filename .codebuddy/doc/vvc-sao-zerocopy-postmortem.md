# VVC SAO Zero-Copy 项目失败复盘

**日期**: 2025-03-27  
**结论**: 项目失败，零收益，代码已清除

---

## 1. 为什么提出一个不可行的目标？

### 根本原因：假设未经证实

| 错误假设 | 现实 |
|---------|------|
| "SAO边界拷贝可以zero-copy，直接从frame引用" | SAO在临时buffer上工作，frame中存储的是未经过SAO的重建值 |
| "边界数据在frame中已准备好" | 边界数据需要在SAO filter**之前**复制到临时buffer |

### 问题追溯

**提出目标时缺少的关键验证：**

1. **没有确认数据来源**：直接引用`frame->data`是否能获取正确的边界数据？
2. **没有确认数据时序**：边界数据在何时可用？
3. **没有画出数据流图**：临时buffer vs frame的关系是什么？

**应该问的问题（第一步就确认）：**
- SAO边界数据从哪里来？
- `sao_extends_edges`的`dst`参数指向哪里？
- `sao_copy_ctb_to_hv`拷贝的数据用途是什么？

## 2. 为什么没有及时发现问题？

### 2.1 代码阅读不深入

```c
// 在sao_extends_edges中：
uint8_t *dst = lc->sao.buffer + y * dst_stride + (x << ps);
```

**没注意到**：`dst`是临时buffer，不是frame。  
**后果**：zero-copy的前提（数据在frame中）根本不成立。

### 2.2 被"简单直觉"误导

- "zero-copy很简单，就是跳过memcpy"
- "边界数据已经在frame里，直接引用就行"

**忽略的事实**：SAO是in-place filter，需要特殊处理边界数据。

### 2.3 没有及时验证

| 应该做 | 实际做 |
|-------|-------|
| **第一步**：检查`frame->data`中是否有正确的SAO边界数据 | 直接写zero-copy代码 |
| **第一步**：确认临时buffer的存在意义 | 假设可以绕过 |
| 写代码前画数据流图 | 凭直觉开始编码 |

---

## 3. 为什么到最后才发现完全不可行？

### 开发流程的错误

```
错误流程：
写代码 -> 调试边界偏移 -> 调试y坐标 -> 启用zero-copy -> fate测试失败 -> 才发现架构限制

正确流程：
确认数据来源 -> 确认数据时序 -> 确认架构可行性 -> 写代码 -> 测试
```

### 关键缺失：架构验证

在写任何代码之前，应该确认：

1. **数据生命周期**：
   - SAO边界数据在何时生成？
   - 在何时被消费？
   - 存储在哪里？

2. **临时buffer的必要性**：
   - 为什么需要`lc->sao.buffer`？
   - 为什么不能直接在frame上操作？

**答案（发现太晚）**：
- `lc->sao.buffer`存储的是经过SAO处理后的CTU数据
- frame中相邻CTU的数据是**未经过SAO**的原始重建值
- 直接引用frame会得到错误数据（CRC mismatch）

## 4. 怎么避免犯同样的错误

### 4.1 强制预检查清单（Go/No-Go）

任何优化项目开始前，必须回答：

```markdown
## 优化项目预检查清单

### 数据来源验证
- [ ] 优化目标的数据从哪里来？
- [ ] 数据是否已经可用（时序）？
- [ ] 数据格式是否符合预期？

### 架构约束检查
- [ ] 目标数据是否可访问（指针/引用）？
- [ ] 是否存在中间层/转换层？
- [ ] 是否有同步/时序依赖？

### 收益/成本估算
- [ ] T_save（预期节省）：____ ms/frame
- [ ] T_cost（新增开销）：____ ms/frame
- [ ] Net gain = T_save - T_cost = ____ ms/frame
- [ ] 需要验证：是否 >= 5%？

### 快速验证
- [ ] 能否用gdb/打印确认数据内容？
- [ ] 能否用最小测试用例验证假设？
- [ ] 是否需要先画出数据流图？

### 决策
- [ ] 所有检查项通过 → 继续
- [ ] 任一检查项不确定 → 暂停，先验证
- [ ] 任一检查项失败 → 停止
```

### 4.2 数据流图优先

**优化前必须画出数据流图：**

```
Before optimizing:
    Source -> [Transformation A] -> [Buffer B] -> [Transformation C] -> Destination
                             ↑
                          数据在这里吗？
                          格式对吗？
                          时序对吗？
```

### 4.3 早期验证策略

| 阶段 | 验证方法 | 通过标准 |
|-----|---------|---------|
| 想法阶段 | 代码走读 + 数据流图 | 确认数据来源和时序 |
| 编码前 | gdb打印关键数据结构 | 确认`frame->data`内容正确 |
| 编码中 | 最小测试用例 | CRC匹配 |
| 编码后 | fate测试 | 全通过 |

### 4.4 "停下来"的信号

出现以下情况必须**立即停止**并重新评估：

1. 发现数据来源与假设不符
2. 发现数据时序问题
3. 发现需要修改架构才能继续
4. 调试时间超过预期2倍

**本项目的停车信号（被忽略）：**
- 调试y坐标偏移时就应该问：为什么需要这么多修正？
- 发现`dst`是临时buffer时就应该停止

## 5. 教训总结

### 技术层面

1. **不要假设，要验证**："数据应该在frame里"是假设，不是事实
2. **画数据流图**：任何涉及buffer的优化，先画数据流图
3. **理解临时buffer的存在意义**：它的存在必有原因，不能轻易绕过

### 流程层面

1. **Go/No-Go检查清单**：必须完成清单才能开始编码
2. **早期验证**：用gdb/打印确认假设，不要等到fate测试
3. **及时止损**：发现架构限制时立即报告，不要试图绕过

### 沟通层面

1. **提出风险**：在发现可能问题时立即报告，不要隐藏
2. **量化收益**：用数据说话，不要用"应该可以"
