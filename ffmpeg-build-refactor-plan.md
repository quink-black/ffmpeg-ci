# FFmpeg 构建系统迁移方案：从 configure + Makefile 到 CMake

## 1. 背景与目标

### 1.1 现有构建系统分析

FFmpeg 当前使用自定义的 `configure` shell 脚本 + `Makefile` 构建系统，核心组成：

| 组件 | 文件 | 职责 |
|------|------|------|
| configure | `configure` (~300KB shell) | 平台检测、特性探测、编译器检测、第三方库检测、生成配置文件 |
| config.h | 构建目录/`config.h` | C 预处理器宏定义（HAVE_xxx, CONFIG_xxx, ARCH_xxx） |
| config_components.h | 构建目录/`config_components.h` | 所有编解码器/复用器/协议等组件的开关宏 |
| config.mak | 构建目录/`ffbuild/config.mak` | Makefile 变量（编译器、标志、路径、特性开关） |
| config.sh | 构建目录/`ffbuild/config.sh` | Shell 变量（供 pkgconfig 生成等脚本使用） |
| 顶层 Makefile | `Makefile` | 汇总所有子目录、定义构建规则 |
| common.mak | `ffbuild/common.mak` | 通用编译规则（.c→.o, .S→.o, .asm→.o 等） |
| library.mak | `ffbuild/library.mak` | 静态库/动态库构建、安装规则 |
| arch.mak | `ffbuild/arch.mak` | 架构特定对象文件收集（NEON, SSE, AVX 等） |
| 各 lib 子目录 Makefile | `libavutil/Makefile` 等 | 定义 OBJS, HEADERS, FFLIBS 等变量 |
| fftools/Makefile | `fftools/Makefile` | 定义 ffmpeg/ffplay/ffprobe 可执行文件的构建 |

### 1.2 迁移目标

- 用 CMake 替代 `configure` + `Makefile`，实现等效的构建能力
- 支持 out-of-source 构建
- 支持平台检测、特性探测、第三方库发现
- 生成与 configure 等价的 `config.h`、`config_components.h`
- 构建 7 个核心库（avutil, avcodec, avformat, avfilter, avdevice, swscale, swresample）
- 构建 3 个工具（ffmpeg, ffplay, ffprobe）
- 保持 FATE 测试兼容
- 支持交叉编译

### 1.3 迁移策略

采用 **渐进式迁移**，CMake 构建系统与现有 configure 系统**并存**，不删除原系统。每步可以通过对比两个系统的输出来验证正确性。

## 2. 整体方案设计

### 2.1 分阶段路线图

```
阶段 1: 基础配置骨架          → 生成 config.h 并与 configure 对比
阶段 2: 最小化编译 libavutil   → 不依赖第三方库，编译核心库
阶段 3: 扩展到所有内置库       → 编译所有 7 个库（无第三方依赖）
阶段 4: 构建命令行工具         → 编译 ffmpeg/ffplay/ffprobe
阶段 5: 第三方库集成           → 逐步添加 x264/x265/aom/dav1d 等
阶段 6: FATE 测试集成          → 运行完整测试套件
阶段 7: 高级特性与收尾         → 交叉编译、安装规则、CI 集成
```

### 2.2 目录结构规划

```
ffmpeg/                          # FFmpeg 源码根目录
├── CMakeLists.txt               # [新增] 顶层 CMake 入口
├── cmake/                       # [新增] CMake 模块目录
│   ├── FFmpegConfig.cmake.in    #   config.h 模板
│   ├── FFmpegComponents.cmake   #   组件（编解码器等）发现与注册
│   ├── FFmpegFeatureDetect.cmake#   平台特性检测（对应 configure 的检测逻辑）
│   ├── FFmpegArch.cmake         #   架构检测（x86/ARM/RISC-V 等）
│   ├── FFmpegCompilerFlags.cmake#   编译器警告/优化标志
│   ├── FFmpegUtils.cmake        #   工具函数
```

## 3. 详细重构步骤

### 阶段 1：基础配置骨架 — 生成 config.h 并与 configure 对比

**目标**：创建 CMake 项目骨架，实现平台检测和特性探测，生成 `config.h`，与 `configure` 生成的版本做 diff 对比。

#### 步骤 1.1：创建顶层 CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.20)
project(FFmpeg VERSION 8.1 LANGUAGES C CXX ASM)

# Options
option(ENABLE_GPL       "Enable GPL licensed code"    OFF)
option(ENABLE_VERSION3  "Enable (L)GPLv3 code"        OFF)
option(ENABLE_NONFREE   "Enable nonfree code"         OFF)
option(ENABLE_SHARED    "Build shared libraries"      OFF)
option(ENABLE_STATIC    "Build static libraries"      ON)
option(ENABLE_DEBUG     "Enable debug symbols"        ON)
option(ENABLE_OPTIMIZATIONS "Enable compiler optimizations" ON)

# Library toggles
option(ENABLE_AVDEVICE   "Build libavdevice"  ON)
option(ENABLE_AVCODEC    "Build libavcodec"   ON)
option(ENABLE_AVFORMAT   "Build libavformat"  ON)
option(ENABLE_AVFILTER   "Build libavfilter"  ON)
option(ENABLE_SWSCALE    "Build libswscale"   ON)
option(ENABLE_SWRESAMPLE "Build libswresample" ON)

# Tool toggles
option(ENABLE_FFMPEG  "Build ffmpeg"  ON)
option(ENABLE_FFPLAY  "Build ffplay"  ON)
option(ENABLE_FFPROBE "Build ffprobe" ON)

# Include CMake modules
list(APPEND CMAKE_MODULE_PATH "${CMAKE_SOURCE_DIR}/cmake")
include(FFmpegArch)
include(FFmpegFeatureDetect)
include(FFmpegCompilerFlags)
include(FFmpegUtils)
```

#### 步骤 1.2：创建架构检测模块 `cmake/FFmpegArch.cmake`

实现对以下架构的检测：
- x86 / x86_64
- aarch64 / arm
- riscv
- mips / loongarch / ppc
- wasm

同时检测 SIMD 指令集（SSE/AVX/NEON/SVE 等）。

#### 步骤 1.3：创建特性检测模块 `cmake/FFmpegFeatureDetect.cmake`

对应 `configure` 中的功能检测，使用 CMake 的 `check_c_source_compiles`、`check_include_file`、`check_function_exists` 等：

- 头文件检测（`HAVE_xxx_H`）
- 函数检测（`HAVE_xxx`）
- 类型大小检测
- 编译器特性检测（atomic, inline asm, thread 等）
- 系统库检测（pthread, zlib, math 等）

#### 步骤 1.4：创建 `cmake/config.h.in` 模板

从现有 `config.h` 反向推导出模板，使用 `#cmakedefine` 和 `#cmakedefine01`：

```c
/* Automatically generated by CMake - do not modify! */
#ifndef FFMPEG_CONFIG_H
#define FFMPEG_CONFIG_H

#define FFMPEG_CONFIGURATION "@FFMPEG_CONFIGURATION@"
#define FFMPEG_LICENSE "@FFMPEG_LICENSE@"
#define CONFIG_THIS_YEAR @CONFIG_THIS_YEAR@

/* Architecture */
#cmakedefine01 ARCH_AARCH64
#cmakedefine01 ARCH_ARM
#cmakedefine01 ARCH_X86
#cmakedefine01 ARCH_X86_32
#cmakedefine01 ARCH_X86_64
/* ... 其余架构和特性宏 ... */
#endif
```

#### 步骤 1.5：生成 config.h 并对比

```cmake
configure_file(
    "${CMAKE_SOURCE_DIR}/cmake/config.h.in"
    "${CMAKE_BINARY_DIR}/config.h"
    @ONLY
)
```

#### 测试方案 1

```bash
# 1. 用 configure 构建，获取参考 config.h
cd /path/to/build_configure
/path/to/ffmpeg/configure --disable-everything --disable-doc
cp config.h config.h.configure_ref

# 2. 用 CMake 构建，获取 CMake 生成的 config.h
mkdir -p /path/to/build_cmake && cd /path/to/build_cmake
cmake /path/to/ffmpeg -DENABLE_OPTIMIZATIONS=OFF
cp config.h config.h.cmake

# 3. 对比两个 config.h
diff config.h.configure_ref config.h.cmake | head -100

# 4. 统计差异数量（目标：逐步减少到 0）
diff config.h.configure_ref config.h.cmake | grep "^[<>]" | wc -l

# 5. 重点检查关键宏：
grep -E "^#define (ARCH_|HAVE_|CONFIG_)" config.h.configure_ref | sort > macros_configure.txt
grep -E "^#define (ARCH_|HAVE_|CONFIG_)" config.h.cmake | sort > macros_cmake.txt
diff macros_configure.txt macros_cmake.txt
```

**验收标准**：
- 所有 `ARCH_*` 宏一致
- 所有 `HAVE_*`（头文件、函数、特性）宏的差异在可接受范围内（平台相关差异除外）
- 关键系统特性宏（如 `HAVE_PTHREADS`, `HAVE_ATOMICS_NATIVE`）一致

### 阶段 2：最小化编译 libavutil — 无第三方依赖

**目标**：成功编译 `libavutil` 静态库，这是所有其他库的基础依赖。

#### 步骤 2.1：创建 `libavutil/CMakeLists.txt`

从现有 `libavutil/Makefile` 提取源文件列表，转换为 CMake：

```cmake
# 基础源文件（无条件编译）
set(AVUTIL_SOURCES
    adler32.c
    aes.c
    aes_ctr.c
    avstring.c
    base64.c
    buffer.c
    channel_layout.c
    cpu.c
    crc.c
    # ... 从 Makefile 中的 OBJS 列表转换
)

# 条件编译源文件
if(HAVE_NEON)
    list(APPEND AVUTIL_SOURCES aarch64/cpu.S aarch64/float_dsp_neon.S ...)
endif()

# 生成版本头文件
configure_file(ffversion.h.in ffversion.h)

add_library(avutil STATIC ${AVUTIL_SOURCES})
target_include_directories(avutil
    PUBLIC  ${CMAKE_SOURCE_DIR} ${CMAKE_BINARY_DIR}
    PRIVATE ${CMAKE_SOURCE_DIR}/compat
)
target_compile_definitions(avutil PRIVATE HAVE_AV_CONFIG_H BUILDING_avutil)
```

#### 步骤 2.2：处理生成文件

- `libavutil/ffversion.h`：通过版本脚本或 CMake 的 `configure_file` 生成
- `config_components.h`：暂时使用 configure 生成的版本，或创建一个最小化版本（全部组件启用）

#### 步骤 2.3：处理架构特定的汇编文件

- ARM NEON: `.S` 文件用 `enable_language(ASM)` + gas 编译
- x86 NASM: 需要 `find_program(NASM nasm)` 并添加自定义命令
- 暂时可以跳过汇编优化文件（`--disable-asm` 等效模式）

#### 测试方案 2

```bash
# 1. CMake 构建 libavutil
mkdir -p build_cmake && cd build_cmake
cmake /path/to/ffmpeg \
    -DCMAKE_BUILD_TYPE=Debug \
    -DENABLE_ASM=OFF          # 先禁用汇编，降低复杂度
make avutil

# 2. 验证静态库生成
ls -la libavutil/libavutil.a
# 应该存在且大小合理 (> 1MB)

# 3. 验证符号表
nm libavutil/libavutil.a | grep "T _av_" | head -20
# 应包含 av_malloc, av_free, av_log 等核心函数

# 4. 与 configure 构建的库对比符号
nm /path/to/build_configure/libavutil/libavutil.a | grep " T " | awk '{print $3}' | sort > syms_configure.txt
nm build_cmake/libavutil/libavutil.a | grep " T " | awk '{print $3}' | sort > syms_cmake.txt
diff syms_configure.txt syms_cmake.txt

# 5. 编写最小测试程序
cat > test_avutil.c << 'EOF'
#include "libavutil/avutil.h"
#include "libavutil/log.h"
#include "libavutil/mem.h"
#include <stdio.h>
int main() {
    printf("avutil version: %s\n", av_version_info());
    void *p = av_malloc(1024);
    if (!p) return 1;
    av_free(p);
    av_log(NULL, AV_LOG_INFO, "avutil CMake build works!\n");
    return 0;
}
EOF
cc -I/path/to/ffmpeg -I build_cmake test_avutil.c -Lbuild_cmake/libavutil -lavutil -lm -lpthread -o test_avutil
./test_avutil
```

**验收标准**：
- `libavutil.a` 成功生成
- 核心符号（`av_malloc`, `av_log`, `av_gettime` 等）存在
- 测试程序能链接并正常运行

### 阶段 3：扩展到所有内置库 — 无第三方依赖

**目标**：编译全部 7 个库，仅使用内置编解码器，禁用所有第三方库。

#### 步骤 3.1：创建各库的 CMakeLists.txt

按依赖顺序实现：

1. **libswresample** → 依赖 avutil
2. **libswscale** → 依赖 avutil
3. **libavcodec** → 依赖 avutil, swresample
4. **libavformat** → 依赖 avutil, avcodec, swresample
5. **libavfilter** → 依赖 avutil, avcodec, avformat, swscale, swresample
6. **libavdevice** → 依赖 avutil, avcodec, avformat, avfilter, swscale, swresample

每个库的 CMakeLists.txt 工作：
- 从对应的 `Makefile` 提取 `OBJS` 和 `OBJS-$(CONFIG_xxx)` 转换为 CMake 条件编译
- 设置 `target_link_libraries` 依赖
- 设置公共头文件

#### 步骤 3.2：实现组件注册系统 `cmake/FFmpegComponents.cmake`

FFmpeg 的组件系统是构建的核心：

```cmake
# 编解码器注册
# 对应 libavcodec/allcodecs.c 中的注册函数
# 需要生成 codec_list.c, parser_list.c, bsf_list.c 等

# 复用器/解复用器注册
# 对应 libavformat 中的 muxer_list.c, demuxer_list.c, protocol_list.c

# 滤镜注册
# 对应 libavfilter 中的 filter_list.c

# 设备注册
# 对应 libavdevice 中的 indev_list.c, outdev_list.c
```

实现策略：
- 初始阶段：使用 configure 生成的组件列表文件作为参考
- 后续：在 CMake 中实现完整的组件选择逻辑

#### 步骤 3.3：处理 config_components.h 生成

`config_components.h` 包含约 2000+ 个 `CONFIG_xxx` 宏，控制每个编解码器、复用器、滤镜等的开关。

初始方案：默认全部启用（等效于 `--enable-everything`），后续实现精细控制。

```cmake
# 生成 config_components.h
# 扫描 libavcodec/allcodecs.c 等文件提取所有组件名
# 为每个组件生成 CONFIG_XXX_ENCODER/DECODER 等宏
```

#### 测试方案 3

```bash
# 1. CMake 构建全部库
mkdir -p build_cmake && cd build_cmake
cmake /path/to/ffmpeg \
    -DCMAKE_BUILD_TYPE=Debug \
    -DENABLE_ASM=OFF
make -j$(nproc)

# 2. 验证所有静态库生成
for lib in avutil swresample swscale avcodec avformat avfilter avdevice; do
    echo -n "lib${lib}: "
    ls -lh lib${lib}/lib${lib}.a 2>/dev/null || echo "MISSING!"
done

# 3. 与 configure 构建对比各库的符号数量
for lib in avutil swresample swscale avcodec avformat avfilter avdevice; do
    cmake_syms=$(nm build_cmake/lib${lib}/lib${lib}.a 2>/dev/null | grep " T " | wc -l)
    conf_syms=$(nm build_configure/lib${lib}/lib${lib}.a 2>/dev/null | grep " T " | wc -l)
    echo "lib${lib}: cmake=${cmake_syms} configure=${conf_syms}"
done

# 4. 验证库间依赖正确性
# libavcodec 不应有未定义的 avutil 符号
nm build_cmake/libavcodec/libavcodec.a | grep " U _av_" | head -10
# 这些 U 符号应该都能在 libavutil.a 中找到

# 5. 构建一个简单的解码测试
cat > test_decode.c << 'EOF'
#include "libavcodec/avcodec.h"
#include <stdio.h>
int main() {
    const AVCodec *codec = avcodec_find_decoder(AV_CODEC_ID_H264);
    if (codec) printf("Found H264 decoder: %s\n", codec->name);
    else printf("H264 decoder not found\n");
    return 0;
}
EOF
# 链接所有库进行测试
cc -I/path/to/ffmpeg -I build_cmake test_decode.c \
    -Lbuild_cmake/libavcodec -lavcodec \
    -Lbuild_cmake/libswresample -lswresample \
    -Lbuild_cmake/libavutil -lavutil \
    -lm -lpthread -lz -o test_decode
./test_decode
```

**验收标准**：
- 所有 7 个静态库成功生成
- 各库符号数量与 configure 构建相差不超过 10%
- 内置编解码器（如 H.264, AAC, MP3）可被 `avcodec_find_decoder` 找到
- 简单解码测试程序能链接并运行

### 阶段 5：第三方库集成

**目标**：逐步添加第三方库支持，按优先级排序。

#### 第三方库集成优先级

| 优先级 | 库名 | 说明 | CMake 发现方式 |
|--------|------|------|----------------|
| P0 | zlib | 基础压缩 | `find_package(ZLIB)` |
| P0 | pthreads | 多线程 | `find_package(Threads)` |
| P1 | openssl | TLS/加密 | `find_package(OpenSSL)` |
| P1 | libxml2 | XML解析 | `find_package(LibXml2)` |
| P1 | x264 | H.264 编码 | `pkg_check_modules(X264 x264)` |
| P1 | x265 | H.265 编码 | `pkg_check_modules(X265 x265)` |
| P1 | libaom | AV1 编解码 | `pkg_check_modules(AOM aom)` |
| P1 | libdav1d | AV1 解码 | `pkg_check_modules(DAV1D dav1d)` |

## 4. 关键技术挑战与解决方案

### 4.1 configure 脚本中的特性检测

**挑战**：`configure` 脚本中包含大量通过编译测试代码片段来检测特性的逻辑（~300KB shell）。

**解决方案**：
- 使用 CMake 的 `CheckCSourceCompiles`, `CheckCSourceRuns`, `CheckIncludeFile` 等模块
- 对于复杂的检测（如内联汇编、compiler built-in），编写自定义的 `check_c_source_compiles` 测试
- 分批实现：先覆盖核心特性，逐步补充边缘情况

### 4.2 组件选择系统

**挑战**：FFmpeg 有 ~2000 个可选组件，每个都有 `CONFIG_XXX` 宏控制。

**解决方案**：
- 编写脚本从 `libavcodec/allcodecs.c`、`libavformat/allformats.c` 等文件中自动提取组件列表
- 在 CMake 中实现 `ENABLE_EVERYTHING` / `DISABLE_EVERYTHING` 快捷选项
- 支持细粒度的 `-DENABLE_DECODER_H264=ON` 式控制

### 4.3 构建生成文件

**挑战**：configure 会生成 `codec_list.c`、`parser_list.c`、`muxer_list.c` 等注册文件。

**解决方案**：
- 在 CMake 中通过 `configure_file` 或自定义 Python/shell 脚本在配置阶段生成这些文件
- 或在构建阶段通过 `add_custom_command` 生成

### 4.4 NASM 汇编编译

**挑战**：CMake 对 NASM 的内置支持有限。

**解决方案**：
- 使用 `add_custom_command` 为每个 `.asm` 文件创建编译规则
- 或使用 `enable_language(ASM_NASM)` + 适当的配置

### 4.5 多平台测试

本机的config.h等配置文件，与本机环境有关，只是单一环境的测试。

- **需要反复对比configure + makefile 与cmake的功能，保证所有功能一致**
- 需要检查cmake中的配置，不能hardcode本机环境、特定配置，如hardcode 链接pthread
- 在其他环境继续完成测试验证（Linux、Windows、Android）

## 6. 核心原则

1. **增量推进**：每一步都是可验证、可回退的
2. **双系统并存**：CMake 系统不影响现有 configure 系统的使用
3. **对比验证**：每个阶段都以 configure 构建结果为参考基准
4. **先跑通再优化**：先实现功能正确性，再追求构建性能和代码优雅
5. **自动化测试**：每个阶段都有明确的测试方案和验收标准
6. **禁止在ffmpeg项目源码中编译**: 在源码树之外编译，防止误提交
7. **使用git add**指定修改的文件路径来提交代码，防止误提交无关文件
8. **禁止修改FFmpeg源码**: 源码一定是正确的，有问题检查ffmpeg configure + makefile是如何处理的，仔细分析代码文件
